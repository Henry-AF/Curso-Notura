# design-sync notes — notura-landing

## Repo shape

This repo is the landing page app itself (`notura-landing`, Next.js App
Router), not a publishable component library — no `dist/`, no
`module`/`main`/`exports['.']`. The sync runs in **package shape, synth-entry
mode** (last resort): the entry is generated by walking `components/`
directly (`cfg.srcDir: "components"` — the `lib/` dir at repo root, used by
the webhook/email code, would otherwise win the default `src/lib/components`
priority order and starve the entry of real files).

`--entry` is passed as `./__no_dist__.js` (a file that doesn't need to
exist) purely so `package-build.mjs`'s PKG_DIR walk lands on the repo root
(where `package.json` has a `name`) instead of `node_modules/notura-landing`
(which doesn't exist, since this package was never installed as a
dependency of itself).

## Fixes applied (forked / config — not the lib/bundle.mjs or lib/emit.mjs core)

- **Tailwind must be precompiled before each build.** `cfg.cssEntry` points
  at `.design-sync/.cache/tailwind-build.css`, regenerated by `cfg.buildCmd`
  (`npx tailwindcss -i ./app/globals.css -o ./.design-sync/.cache/tailwind-build.css`).
  The converter never runs Tailwind itself — always re-run `buildCmd` before
  `package-build.mjs` if any component's classNames changed, or styling will
  silently go stale.
- **`next/image` / `next/link` are shimmed away** via `cfg.tsconfig` pointing
  at `.design-sync/tsconfig.sync.json` (NOT the repo's real `tsconfig.json`)
  with `paths` aliasing both to `.design-sync/shims/next-image.tsx` /
  `next-link.tsx`. Without this, esbuild bundles Next's real client runtime,
  which reads `process.env.__NEXT_*` at module scope and throws
  `ReferenceError: process is not defined` in the browser — killing the
  ENTIRE bundle's IIFE evaluation (nothing reaches `window.NoturaLanding`).
- **`.design-sync/overrides/source-kit.mjs`** (declared in `cfg.libOverrides`)
  patches synth-entry generation: plain `export * from <file>` never
  re-exports default exports (ES module spec), so `ShinyText` and `TextType`
  (both `export default`) never reached the bundle. The fork adds an
  `export { default as <Base> } from <file>` line per file that actually has
  a default export. **If this fork's `lib/source-kit.mjs` upstream changes,
  diff and re-merge the entry-generation block** (search for `comps.map`).
  Needs the node_modules junction: `New-Item -ItemType Junction
  -Path .design-sync/node_modules -Target ../.ds-sync/node_modules`
  (PowerShell; `ln -sfn` silently creates a real empty dir on this Windows
  box instead of a symlink — always verify with `Get-Item ... | Select
  LinkType` after creating it, don't trust exit code 0).
- **`componentSrcMap` excludes** 10 tiny legal-page typographic helpers
  (`CrossLinks`, `Fill`, `LegalPage`, `Note`, `P`, `Table`, `Td`, `Th`,
  `Thead`, `Tr` — all named exports from `components/legal/LegalContent.tsx`)
  and `ClarityProvider` (analytics side-effect component, renders nothing).

## AuthorityBar capture — unresolved tooling issue

`package-capture.mjs`'s `?story=Default` navigation for `AuthorityBar`
times out (`net::ERR_ABORTED` on the document request itself, then
`page.goto` 20s timeout) — **reproduced 5/5 times**, both via the real
script and ad-hoc Playwright repros, isolated to this one component. Root
cause NOT found (server serves the file fine via plain `fetch`;
`package-validate.mjs`'s own render-check — same `networkidle` navigation
strategy, just without the `?story=` query — renders it cleanly; the
compiled bundle/preview JS is byte-for-byte the same code path other
components use successfully). Graded `good` for `AuthorityBar.grade.json`
manually from `_screenshots/sections__AuthorityBar.png` (validate's own
render-check screenshot) instead of a capture review sheet — visually
confirmed correct (styled stats bar, real numbers, dividers).

**Re-sync risk:** because capture never succeeded, no `sourceKey`/`gradeKey`
bookkeeping was ever written to `.design-sync/.cache/review/AuthorityBar.json`
(only the hand-written `.grade.json` exists). The next sync's diff will NOT
recognize this as carried-forward and will likely re-attempt capture. If it
fails again, re-grade from the validate render-check screenshot the same way
rather than burning more time chasing the capture hang — the component
itself is fine.

## Preview scope

14 of 26 components got authored rich previews (the most representative /
reused ones): GrainientBtn, ShinyText, ShuffleText, TextType, StaggeredMenu,
TextReveal, CountUp, InfiniteMarquee, MagneticButton, SpotlightCard,
TiltCard, Navbar, Footer, AuthorityBar. The other 12 (Aurora, Particles,
SplitText, ComboSection, FAQSection, Hero, NewsletterSection, OfferSection,
PricingSection, ProblemSection, SolutionSection, TestimonialsSection) ship
on the floor card / their own no-props render (all render real content
already, just unauthored — fine to promote on any future sync; not a
failure state).

`StaggeredMenu`'s expanded/open panel state is interaction-only (click) and
isn't captured — only the closed toggle button is graded. `MagneticButton`,
`SpotlightCard`, `TiltCard` similarly only show their static rest state
(magnetic pull / spotlight glow / tilt are hover-only).

## Re-sync risks (forward-looking)

- **`cfg.tsconfig` and `cfg.cssEntry` both point inside `.design-sync/` to
  generated/hand-written files, not the repo's real `tsconfig.json` /
  `app/globals.css`.** A future sync must NOT "simplify" these back to the
  real files — that reintroduces the `process is not defined` crash and the
  unstyled-Tailwind regression respectively.
- **The Tailwind precompile (`buildCmd`) scans the CURRENT `components/**`
  tree via `tailwind.config.ts`'s content globs.** If new components are
  added with classNames not yet used anywhere at compile time, re-run
  `buildCmd` before `package-build.mjs` — it's not automatic.
- **The `.design-sync/overrides/source-kit.mjs` fork is load-bearing for 2 of
  26 components (ShinyText, TextType).** If a future design-sync skill
  update changes `lib/source-kit.mjs`'s entry-generation contract, this fork
  needs re-diffing, not blind deletion — deleting it without fixing the
  default-export gap silently drops those 2 components from the bundle
  again (no build error, just absent from `window.NoturaLanding`).
- **AuthorityBar's capture hang (above) is unexplained.** If it starts
  affecting other components too on a future sync, that would point to a
  real environment issue (proxy/AV/Chrome) worth escalating instead of
  hand-grading around it again.
